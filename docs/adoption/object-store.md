# 3tears-object-store

`threetears.object_store` -- streaming S3-compatible object store for large
binary artifacts.

## Problem

Large binary artifacts -- pcaps, DB dumps, rendered reports, evidence files
-- don't belong in a relational table, and naive S3 client usage tends to
buffer the whole object in memory on upload or download, which fails at
scale for multi-GB files.

## What it does

- Implements the dependency-free `ObjectStore` protocol from
  `3tears-media-contracts` over any S3-compatible backend (MinIO in dev, S3
  in prod).
- Streaming by contract: uploads move through one part-size buffer at a
  time via S3 multipart; downloads yield the response body in chunks.
- Keys are generated by `keys.build_object_key()` in a fixed, tenant-first
  layout: `<customer_id>/<scope>/<category>/<YYYY>/<MM>/<DD>/<object_id>/<filename>`,
  so objects are namespaced per tenant and browsable by date without a
  separate index.

## Design philosophy

"Streaming by contract" means the streaming behavior is part of the
interface guarantee, not an implementation detail that could regress -- a
multi-GB object never has to sit whole in a pod's memory, on either the
read or write path. It was generalized from an internal service's ad hoc S3
wrapper into this streaming, contract-conformant implementation.

## When to adopt

Any app that needs to store large binary artifacts (backups, evidence,
reports, media) in a way that scales past what fits comfortably in memory.

## Composes with

- [`core`](core.md) -- pulled in for its secret-reference resolver.
- [`media-contracts`](media-contracts.md) -- the `ObjectStore` protocol it
  implements.
- [`backup`](backup.md) -- a typical consumer.

## Install

```bash
pip install 3tears-object-store
```
